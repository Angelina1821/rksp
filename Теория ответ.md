# Ответы на билет №2
## 1.	OpenAPI как контракт между командами и системами

OpenAPI это спецификация для описания REST API в формате YAML или JSON, раньше она называлась Swagger. По сути это контракт между командами, например бекендом и фронтендом, потому что в этом файле описанно как должны выглядеть запросы и ответы. Без такого контракта команды могут работать по разной документации, API может меняться когда захочется и никто об этом не узнает, а фронтенд не может начать разработку пока бекенд не готов. OpenAPI решает эти проблемы: он становится единым источником правды, по нему можно сгенерировать клиентский SDK или серверную заглушку, а также автоматически получить Swagger UI документацию. В итоге разработка идет параллельно и ошибок на стыке меньше

## 2.	Kafka consumer: жизненный цикл и обработка сообщений.

Жизненный цикл Kafka consumer состоит из нескольких этапов. Сначала мы создаем consumer с настройками: адрес сервера, group.id, десириализаторы и параметр auto.offset.reset который говорит с какого места читать сообщения. Потом consumer подписывается на топик или несколько топиков через subscribe. Дальше начинается бесконечный цикл, в котором мы вызываем poll. Когда сообщения получены, мы их обрабатываем и самое важное это фиксируем offset, то есть запоминаем какое сообщение уже обработано. Offset можно фиксировать автоматически или вручную. Если consumer упадет, он при перезапуске продолжит чтение с последнего зафиксированного offset. Также есть механизм rebalance - это когда партиции перераспределяются между consumer в группе, например если добавился новый consumer

## 3.	Java Memory Model: зачем она нужна разработчику.

Java Memory Model это спецификация которая описывает как потоки взаимодействуют с памятью в Java. Она нужна разработчику потому что без нее можно наделать много багов. Основные проблемы которые решает JMM - это видимость, переупорядочивание и атомарность. Видимость означает что изменения в одном потоке могут быть не видны другому. Переупорядочивание - это когда компилятор или процессор меняют порядок инструкций для оптимизации, и в многопоточном коде это может дать неожиданный результат. Атомарность - например операция инкремента на самом деле не атомарная, она состоит из трех шагов. JMM дает нам правила happens-before, то есть гарантии когда изменения становятся видны. Без понимания JMM очень сложно отлаживать многопоточные приложения, потому что баги могут воспроизводиться не всегда.
